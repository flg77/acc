# ACC Security Hardening Guide

This document describes the complete ACC security architecture across all phases. Phases 0a and 0b are **implemented**. Phases 0c through 4 are **planned** ‚Äî this document captures the design so that every stakeholder understands the full intended security posture before implementation begins.

---

The table below shows the current state of each security control:

| # | Gap | Severity | Status | Phase |
|---|-----|----------|--------|-------|
| G-1 | `agent_id` is a random UUID ‚Äî no stable identity across restarts | High | Planned | Phase 2 |
| G-2 | NATS anonymous auth ‚Äî any pod can publish/subscribe to `acc.>` | Critical | Partial (Phase 0c planned) | Phase 0c |

## Governance Layer (Rego Rules ‚Äî Implemented)

ACC enforces three governance tiers through OPA Rego policies. These policies are live in `regulatory_layer/` and evaluated by agent processes and/or an OPA bundle server.

### Category A ‚Äî Constitutional Rules (Immutable)

Located in `regulatory_layer/constitutional_rhoai.rego`. These rules are compiled to a WASM blob (`category_a.wasm`) and loaded into each agent process. They cannot be changed at runtime.

| Rule | Description |
|------|-------------|
| **A-001** | Signals may only be published to the agent's own `collective_id` namespace ‚Äî no cross-namespace publishing without bridge registration |
| **A-002** | Category A rules are themselves immutable ‚Äî no runtime modification of the constitutional layer |
| **A-003** | `TERMINATE` signal may only be published by an agent with `role=arbiter` |
| **A-004** | All outbound signals must carry a non-empty Ed25519 signature field (presence enforced; cryptographic verify in Phase 0a) |
| **A-005** | Every agent must publish a HEARTBEAT within `heartbeat_interval_s`; arbiter escalates absent agents |
| **A-006** | No agent may escalate its own capabilities or request a higher governance role |
| **A-007** | LLM output submitted to the bus must be valid JSON |
| **A-008** | Agents must request human review after `max_generation = 10` consecutive LLM calls without arbiter approval |
| **A-009** | Audit trail entries are immutable ‚Äî no retroactive deletion or modification |
| **A-010** | Cross-collective communication (bridge delegation) requires explicit `bridge_enabled=true` registration |

### Category B ‚Äî Live-Updatable Setpoints

Located in `regulatory_layer/conditional_rhoai.rego` with values in `regulatory_layer/data_rhoai.json`. These are pushed to agents via the OPA bundle server and hot-reload without pod restarts.

| Rule | Default setpoint | Description |
|------|-----------------|-------------|
| **B-001** | `min_health_for_sync: 0.70` | Memory sync is rejected when collective health drops below 70% |
| **B-002** | ‚Äî | Task assignments are rejected when an agent is in DEGRADED state |
| **B-003** | `max_drift_score: 0.35` | Agents exceeding this drift score from their role centroid trigger rogue detection |
| **B-004** | `max_llm_calls_per_minute: 20` | LLM rate limit per agent; overridable per-role via `category_b_overrides` |
| **B-005** | `memory_sync_batch_size: 50` | Adaptive batch size for LanceDB sync operations |
| **B-006** | Varies by role | Maximum task duration before timeout (role-specific) |
| **B-007** | `min_quorum_agents: 2` | Collective is non-functional below this active agent count |
| **B-008** | `max_gpu_hours_per_day: 24` | GPU budget enforcement (RHOAI mode only) |

Per-role overrides are applied via `role_definition.category_b_overrides` in the agent's role definition. Example:
```yaml
category_b_overrides:
  token_budget: 4096        # overrides default for this role
  rate_limit_rpm: 30
```

### Category C ‚Äî Adaptive Rules (Arbiter-Signed)

Located in `regulatory_layer/adaptive_rhoai.rego`. These are generated by the arbiter from recurring ICL episode patterns and cryptographically signed before being applied to agents.

| Rule | Description |
|------|-------------|
| **C-AUTO-001** | PDF task gating: only assign to agents with >40% resource headroom |
| **C-AUTO-002** | Financial analysis: only route to analysts with `health_score ‚â• 0.8` |
| **C-AUTO-003** | vLLM throttling: halt new LLM calls when queue depth > 8 |
| **C-AUTO-004** | Bulk sync scheduling: defer memory sync to off-peak hours (outside 09:00‚Äì17:00 UTC) |

New Cat-C rules are generated by the arbiter and signed with its Ed25519 private key. Agents verify the signature before applying (Phase 0a). The arbiter enforces a confidence threshold (`spec.governance.categoryC.confidenceThreshold`) before signing any new rule.

## Phase 0a ‚Äî Ed25519 Signature Verification ‚úÖ Implemented

**What it does:** All `ROLE_UPDATE` payloads on NATS must carry a valid Ed25519 signature from the arbiter. `RoleStore.apply_update()` cryptographically verifies the signature against the arbiter's public key before applying any role change.

**Key files:**
- `acc/role_store.py` ‚Äî `apply_update()` uses `cryptography.hazmat.primitives.asymmetric.ed25519`
- `acc/config.py` ‚Äî `SecurityConfig.arbiter_verify_key: str` (Base64 raw 32-byte public key)
- `acc-config.yaml` ‚Äî `security.arbiter_verify_key` field

**Configuration:**
```bash
# Generate key pair (see docs/howto-role-infusion.md for the full script)
export ACC_ARBITER_VERIFY_KEY=<base64-encoded-32-byte-ed25519-public-key>
```

**What it does NOT do (yet):**
- Does not verify signatures on individual NATS bus signals (only ROLE_UPDATE)
- Does not use a stable cryptographic agent identity (Phase 2)

---

## Phase 0b ‚Äî Redis Authentication ‚úÖ Implemented

**What it does:** Redis working memory requires a password. Agents pass `password=` to `redis.asyncio.from_url()`. The operator generates an `acc-redis-auth` Secret and injects it into all agent pods.

**Key files:**
- `acc/config.py` ‚Äî `WorkingMemoryConfig.password: str`
- `acc/backends/working_memory_redis.py` ‚Äî `_build_redis_client()` factory
- `acc-config.yaml` ‚Äî `working_memory.url` and `working_memory.password`

**Configuration:**
```bash
export ACC_REDIS_URL=redis://localhost:6379
export ACC_REDIS_PASSWORD=$(openssl rand -hex 32)
```

**What it does NOT do (yet):**
- No TLS on Redis (Phase 2)
- No per-role ACL (Phase 2)

## Phase 0c ‚Äî NATS NKeys Authentication üî≤ Planned

**Problem:** NATS currently accepts anonymous connections. Any pod that can reach port 4222 can subscribe to `acc.>` and read all agent signals, or publish unauthorized signals.

**Solution:** NATS NKeys (NaCl keypairs) provide per-user cryptographic authentication. Each agent role gets a unique NKey seed; the NATS server config lists the corresponding public keys. Agents connect with `nats.connect(nkeys_seed=...)`.

**NKey permission matrix** (once implemented):

| Role | Publish | Subscribe |
|------|---------|-----------|
| `ingester` | `acc.{cid}.task`, `acc.{cid}.heartbeat`, `acc.{cid}.register`, `acc.bridge.{cid}.*.delegate` | `acc.{cid}.task` |
| `analyst` | `acc.{cid}.task`, `acc.{cid}.heartbeat`, `acc.{cid}.alert`, `acc.bridge.{cid}.*.delegate` | `acc.{cid}.task`, `acc.bridge.*.{cid}.result` |
| `arbiter` | `acc.{cid}.heartbeat`, `acc.{cid}.role_approval` | `acc.{cid}.role_update`, `acc.{cid}.alert` |
| `synthesizer` | `acc.{cid}.task`, `acc.{cid}.heartbeat`, `acc.bridge.{cid}.*.delegate` | `acc.{cid}.task`, `acc.bridge.*.{cid}.result` |
| `observer` | `acc.{cid}.heartbeat` | `acc.{cid}.>` (read-only) |

**Files to change:**
- `acc/config.py` ‚Äî Add `ACC_NATS_NKEY_SEED` to `_ENV_MAP`
- `acc/agent.py` ‚Äî Pass `nkeys_seed=` to `nats.connect()`
- `operator/internal/reconcilers/infra/nats.go` ‚Äî Render per-role NKey user config into NATS ConfigMap
- `operator/internal/reconcilers/collective/agent_deployment.go` ‚Äî Mount NKey Secret into pods

**Note:** Phase 0c can only be properly scoped now that ACC-9 (cross-collective bridge) has defined the full bridge subject schema (`acc.bridge.{from}.{to}.delegate`, etc.). The permission matrix above includes bridge subjects.

## Phase 1 ‚Äî Cilium L4/L7 NetworkPolicy üî≤ Planned

**Problem:** Standard Kubernetes `NetworkPolicy` objects provide L4 (IP + port) isolation only. There is no enforcement at the NATS protocol level.

**Solution:** Cilium replaces kube-proxy with an eBPF data plane that supports L7-aware policies and per-pod egress control with FQDN matching.

**Prerequisite:** Cilium must be installed in the cluster. The operator detects it via the `cilium.io/v2` API group. If absent, the operator falls back to standard `NetworkPolicy` and emits a `NetworkPolicyReady=False` condition.

**Planned policies (per component):**

```yaml
# Agent pods ‚Äî ingress from same collective only; egress to NATS + Redis + LLM
egress:
  - toEndpoints:
      - matchLabels: {acc.redhat.io/component: nats}
    toPorts: [{port: "4222"}]
  - toEndpoints:
      - matchLabels: {acc.redhat.io/component: redis}
    toPorts: [{port: "6379"}]
  - toFQDNs:
      - matchPattern: "*.anthropic.com"    # Anthropic LLM backend
  - toEndpoints:
      - matchLabels: {acc.redhat.io/component: ollama}
    toPorts: [{port: "11434"}]
```

**Edge mode ‚Äî NATS leaf port 7422:** Edge agent pods need an additional egress rule to the hub's leaf node port:
```yaml
  - toCIDR: [{cidr: "0.0.0.0/0"}]   # hub may be external
    toPorts: [{port: "7422"}]         # NATS leaf protocol
```

**Files to create:**
- `operator/internal/reconcilers/security/cilium.go` ‚Äî New sub-reconciler; creates `CiliumNetworkPolicy` CRs
- `operator/internal/util/discovery.go` ‚Äî Add Cilium API group detection
- `operator/api/v1alpha1/agentcorpus_types.go` ‚Äî Add `SecuritySpec` struct

**Edge mode behavior:** Cilium policies are NOT deployed in edge mode. Edge uses standard `NetworkPolicy` only (no eBPF at MicroShift by default).

## Phase 2 ‚Äî SPIFFE/SPIRE mTLS üî≤ Planned

**Problem:** Agent identity is a random UUID generated at startup. An agent that restarts gets a new identity. There is no cryptographic proof that a pod claiming to be `analyst` in `sol-01` actually is.

**Solution:** SPIFFE/SPIRE provides stable X.509 workload identity (SVIDs) with automatic rotation. Each agent pod receives a short-lived certificate with a SPIFFE URI encoding its corpus, collective, role, and stable agent ID suffix.

**SPIFFE ID format:**
```
spiffe://{trust_domain}/acc/{corpus_name}/{collective_id}/{role}/{agent_id_suffix}
```
Example: `spiffe://acc.corp/acc/my-corpus/sol-01/analyst/9c1d`

**What SPIRE provides:**
- SPIRE Server (cluster-scoped): issues SVIDs, stores registration entries, talks to K8s attestor
- SPIRE Agent (DaemonSet): node-level workload attestor, serves Workload API socket
- Registration entries: one per `(namespace, serviceAccountName, podLabel)` tuple
- SVIDs rotate automatically (default 1-hour TTL)

**Stable `agent_id`:** The operator stamps `acc.redhat.io/agent-id: {role}-{4-char-hex}` on each agent Deployment. Agents read `ACC_AGENT_ID` from a `fieldRef` label injection. This gives stable identity across pod restarts ‚Äî a breaking change from the current random UUID.

**NATS TLS with SVID:**
```
tls {
  cert_file: /run/spiffe/tls.crt
  key_file:  /run/spiffe/tls.key
  ca_file:   /run/spiffe/bundle.crt
  verify:    true
}
```

**Redis mTLS:**
- Redis config: `tls-port 6380`; agents connect on port 6380 with SVID client cert
- Adds per-role ACL (`requirepass` stays on port 6379 for backward compat)

**Files to create:**
- `operator/internal/reconcilers/security/spire.go` ‚Äî Creates `SpiffeID` CRs or calls SPIRE Server API
- `operator/internal/reconcilers/infra/nats.go` ‚Äî TLS config generation (extend existing file)
- `operator/internal/reconcilers/infra/redis.go` ‚Äî TLS config + port 6380 (extend existing file)
- `operator/internal/reconcilers/collective/agent_deployment.go` ‚Äî `spiffe-helper` sidecar injection
- `acc/agent.py` ‚Äî Read stable `ACC_AGENT_ID`; mTLS NATS connect

**Prerequisite:** SPIRE Server + DaemonSet installed. The operator detects via `spire.spiffe.io/v1alpha1` API group. If absent, `SpireReady=False` condition is set; operator falls back to NKeys only (Phase 0c).

**SPIRE is NOT installed by the ACC operator** ‚Äî it is a cluster prerequisite.

**Edge mode:** SPIRE is not available at edge. Edge uses NKeys (Phase 0c) + manual TLS certificates via `spiffe-helper` with a self-signed CA (Phase 4).

## Phase 3 ‚Äî Tetragon Kernel-Level Cat-A üî≤ Planned

**Problem:** The Cat-A WASM OPA evaluator currently returns `True` unconditionally (`_cat_a_allow = True` in `acc/governance.py`). Real Cat-A enforcement requires actual WASM evaluation and, for production clusters, kernel-level observability that application code cannot bypass.

**Solution:** Tetragon attaches eBPF programs to kernel hooks (`execve`, `connect`, file `openat`) and emits structured `ProcessEvent` JSON on a gRPC stream. `CognitiveCore` subscribes to this stream and includes kernel events in Cat-A evaluation alongside real WASM OPA evaluation.

**TracingPolicy targets (planned):**

| Hook | Event | Cat-A response |
|------|-------|----------------|
| `execve` | Unexpected binary execution (outside known paths) | `ALERT_ESCALATE` Cat-A; observe-only in first 4 weeks |
| `connect` | Outbound TCP to non-approved CIDRs | `ALERT_ESCALATE` Cat-A |
| `openat` | `/proc/*/mem` reads by agent process | `ALERT_ESCALATE` Cat-A; potential kill (Phase 3 enforce mode) |

**Observe-only first:** `ACC_TETRAGON_ENFORCE=false` (default). Tetragon runs for 4 weeks logging events without triggering Cat-A responses. Once a baseline is established, `ACC_TETRAGON_ENFORCE=true` gates Cat-A decisions on kernel events.

**Files to create:**
- `operator/internal/reconcilers/security/tetragon.go` ‚Äî Creates `TracingPolicy` CRs
- `acc/tetragon_bridge.py` ‚Äî gRPC stream ‚Üí NATS `ALERT_ESCALATE` bridge
- `acc/cognitive_core.py` ‚Äî Replace `_cat_a_allow = True` with real WASM + Tetragon event evaluation
- `acc/config.py` ‚Äî Add `ACC_TETRAGON_ENFORCE`, `ACC_TETRAGON_GRPC_ADDR`

**Prerequisite:** Tetragon installed (operator detects via `cilium.io/v1alpha1 TracingPolicy` CRD).

**Edge mode:** Tetragon is NOT available at edge. eBPF programs require kernel access that is not available in standard MicroShift / Podman container environments. Cat-A falls back to WASM-only at edge.

**What Tetragon is NOT used for:** NATS subject-level ACL enforcement. Tetragon sees raw TCP bytes, not NATS protocol messages. Use NATS NKeys (Phase 0c) for subject ACLs.

## Phase 4 ‚Äî Hardened Standalone (Podman) üî≤ Planned

**Problem:** Phases 1‚Äì3 are Kubernetes-only. The standalone Podman mode has a separate hardening track.

**What is already done:**
- Redis password via `.env` (`ACC_REDIS_PASSWORD`) ‚Äî implemented in Phase 0b
- Ed25519 ROLE_UPDATE verification ‚Äî implemented in Phase 0a

**What is planned:**

| Control | Status | Notes |
|---------|--------|-------|
| NATS NKeys | Planned | Same NKey seed approach as Phase 0c; `nats.conf` generated from `.env` file |
| Redis `requirepass` + TLS | Planned | Self-signed CA; `tls-port 6380`; spiffe-helper binary |
| Manual X.509 certificates | Planned | `spiffe-helper` + self-signed CA for mTLS between NATS, Redis, agents |
| Tetragon | **Not available** | eBPF not attachable in standard Podman containers |
| Cilium | **Not available** | Requires Kubernetes |
| SPIRE | **Not available** | Manual `spiffe-helper` + self-signed CA instead |

**Files to change (Phase 4):**
- `deploy/podman-compose.yml` ‚Äî NKey + Redis TLS volumes
- `deploy/Containerfile.redis` ‚Äî `tls-port 6380`, `requirepass ${REDIS_PASSWORD}`, `tls-cert-file`
- `deploy/podman-compose.yml` ‚Äî `spiffe-helper` sidecar pattern with shared volume
- `acc-config.yaml.example` ‚Äî Document all new security fields

## Implementation Sequence

The dependencies between phases are:

```
Phase 0a ‚Äî Ed25519 ROLE_UPDATE verify    ‚Üê DONE
Phase 0b ‚Äî Redis password auth           ‚Üê DONE
Phase 0c ‚Äî NATS NKeys                   ‚Üê start next (depends on ACC-9 bridge
‚îÇ                                          subject schema ‚Äî now available)
‚îÇ
‚îú‚îÄ‚îÄ Phase 1 ‚Äî Cilium NetworkPolicy       ‚Üê depends on ACC-9 (bridge subjects ‚Üí
‚îÇ                                          port 7422 egress rule for edge)
‚îÇ
‚îú‚îÄ‚îÄ Phase 2 ‚Äî SPIFFE/SPIRE mTLS          ‚Üê depends on Phase 0c (NKeys ‚Üí SPIRE
‚îÇ     ‚îÇ        stable agent_id           ‚îÇ  replaces NKeys progressively)
‚îÇ     ‚îî‚îÄ‚îÄ Phase 4 ‚Äî Hardened Standalone  ‚Üê parallel; same concepts, no SPIRE
‚îÇ
‚îî‚îÄ‚îÄ Phase 3 ‚Äî Tetragon Cat-A             ‚Üê parallel with Phase 2;
                                           requires Tetragon installed
```

**Recommended order for next work:**
1. **Phase 0c (NATS NKeys)** ‚Äî highest risk reduction per effort; no infrastructure prerequisites; ~1 week
2. **Phase 1 (Cilium NetworkPolicy)** ‚Äî standard `NetworkPolicy` fallback makes this low-risk; ~2 weeks
3. **Phase 4 (Standalone hardening)** ‚Äî parallel with Phase 1; self-signed CA setup
4. **Phase 2 (SPIRE)** ‚Äî most complex; ~5 weeks; requires SPIRE Server installed
5. **Phase 3 (Tetragon)** ‚Äî final; ~5 weeks; observe-only mode first
