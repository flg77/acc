# ACC Security Hardening Guide

This document describes the complete ACC security architecture across all phases. Phases 0a and 0b are **implemented**. Phases 0c through 4 are **planned** — this document captures the design so that every stakeholder understands the full intended security posture before implementation begins.

---

## Security Gap Inventory

The table below shows the current state of each security control:

| # | Gap | Severity | Status | Phase |
|---|-----|----------|--------|-------|
| G-1 | `agent_id` is a random UUID — no stable identity across restarts | High | Planned | Phase 2 |
| G-2 | NATS anonymous auth — any pod can publish/subscribe to `acc.>` | Critical | Partial (Phase 0c planned) | Phase 0c |
| G-3 | No mTLS between NATS, Redis, and agents | High | Planned | Phase 2 |
| G-4 | Ed25519 signature verification in `RoleStore.apply_update()` | High | **Implemented** (Phase 0a) | — |
| G-5 | Redis unauthenticated — any pod reaching Redis can read/write agent state | High | **Implemented** (Phase 0b) | — |
| G-6 | Cat-A WASM placeholder — `_cat_a_allow = True` hardcoded | Critical | Planned | Phase 3 |
| G-7 | No network policy — agents can reach any cluster endpoint | High | Planned | Phase 1 |

## Governance Layer (Rego Rules — Implemented)

ACC enforces three governance tiers through OPA Rego policies. These policies are live in `regulatory_layer/` and evaluated by agent processes and/or an OPA bundle server.

### Category A — Constitutional Rules (Immutable)

Located in `regulatory_layer/constitutional_rhoai.rego`. These rules are compiled to a WASM blob (`category_a.wasm`) and loaded into each agent process. They cannot be changed at runtime.

| Rule | Description |
|------|-------------|
| **A-001** | Signals may only be published to the agent's own `collective_id` namespace — no cross-namespace publishing without bridge registration |
| **A-002** | Category A rules are themselves immutable — no runtime modification of the constitutional layer |
| **A-003** | `TERMINATE` signal may only be published by an agent with `role=arbiter` |
| **A-004** | All outbound signals must carry a non-empty Ed25519 signature field (presence enforced; cryptographic verify in Phase 0a) |
| **A-005** | Every agent must publish a HEARTBEAT within `heartbeat_interval_s`; arbiter escalates absent agents |
| **A-006** | No agent may escalate its own capabilities or request a higher governance role |
| **A-007** | LLM output submitted to the bus must be valid JSON |
| **A-008** | Agents must request human review after `max_generation = 10` consecutive LLM calls without arbiter approval |
| **A-009** | Audit trail entries are immutable — no retroactive deletion or modification |
| **A-010** | Cross-collective communication (bridge delegation) requires explicit `bridge_enabled=true` registration |

### Category B — Live-Updatable Setpoints

Located in `regulatory_layer/conditional_rhoai.rego` with values in `regulatory_layer/data_rhoai.json`. These are pushed to agents via the OPA bundle server and hot-reload without pod restarts.

| Rule | Default setpoint | Description |
|------|-----------------|-------------|
| **B-001** | `min_health_for_sync: 0.70` | Memory sync is rejected when collective health drops below 70% |
| **B-002** | — | Task assignments are rejected when an agent is in DEGRADED state |
| **B-003** | `max_drift_score: 0.35` | Agents exceeding this drift score from their role centroid trigger rogue detection |
| **B-004** | `max_llm_calls_per_minute: 20` | LLM rate limit per agent; overridable per-role via `category_b_overrides` |
| **B-005** | `memory_sync_batch_size: 50` | Adaptive batch size for LanceDB sync operations |
| **B-006** | Varies by role | Maximum task duration before timeout (role-specific) |
| **B-007** | `min_quorum_agents: 2` | Collective is non-functional below this active agent count |
| **B-008** | `max_gpu_hours_per_day: 24` | GPU budget enforcement (RHOAI mode only) |

Per-role overrides are applied via `role_definition.category_b_overrides` in the agent's role definition. Example:
```yaml
category_b_overrides:
  token_budget: 4096        # overrides default for this role
  rate_limit_rpm: 30
```

### Category C — Adaptive Rules (Arbiter-Signed)

Located in `regulatory_layer/adaptive_rhoai.rego`. These are generated by the arbiter from recurring ICL episode patterns and cryptographically signed before being applied to agents.

| Rule | Description |
|------|-------------|
| **C-AUTO-001** | PDF task gating: only assign to agents with >40% resource headroom |
| **C-AUTO-002** | Financial analysis: only route to analysts with `health_score ≥ 0.8` |
| **C-AUTO-003** | vLLM throttling: halt new LLM calls when queue depth > 8 |
| **C-AUTO-004** | Bulk sync scheduling: defer memory sync to off-peak hours (outside 09:00–17:00 UTC) |

New Cat-C rules are generated by the arbiter and signed with its Ed25519 private key. Agents verify the signature before applying (Phase 0a). The arbiter enforces a confidence threshold (`spec.governance.categoryC.confidenceThreshold`) before signing any new rule.

## Phase 0a — Ed25519 Signature Verification ✅ Implemented

**What it does:** All `ROLE_UPDATE` payloads on NATS must carry a valid Ed25519 signature from the arbiter. `RoleStore.apply_update()` cryptographically verifies the signature against the arbiter's public key before applying any role change.

**Key files:**
- `acc/role_store.py` — `apply_update()` uses `cryptography.hazmat.primitives.asymmetric.ed25519`
- `acc/config.py` — `SecurityConfig.arbiter_verify_key: str` (Base64 raw 32-byte public key)
- `acc-config.yaml` — `security.arbiter_verify_key` field

**Configuration:**
```bash
# Generate key pair (see docs/howto-role-infusion.md for the full script)
export ACC_ARBITER_VERIFY_KEY=<base64-encoded-32-byte-ed25519-public-key>
```

**What it does NOT do (yet):**
- Does not verify signatures on individual NATS bus signals (only ROLE_UPDATE)
- Does not use a stable cryptographic agent identity (Phase 2)

## Phase 0b — Redis Authentication ✅ Implemented

**What it does:** Redis working memory requires a password. Agents pass `password=` to `redis.asyncio.from_url()`. The operator generates an `acc-redis-auth` Secret and injects it into all agent pods.

**Key files:**
- `acc/config.py` — `WorkingMemoryConfig.password: str`
- `acc/backends/working_memory_redis.py` — `_build_redis_client()` factory
- `acc-config.yaml` — `working_memory.url` and `working_memory.password`

**Configuration:**
```bash
export ACC_REDIS_URL=redis://localhost:6379
export ACC_REDIS_PASSWORD=$(openssl rand -hex 32)
```

**What it does NOT do (yet):**
- No TLS on Redis (Phase 2)
- No per-role ACL (Phase 2)

## Phase 0c — NATS NKeys Authentication 🔲 Planned

**Problem:** NATS currently accepts anonymous connections. Any pod that can reach port 4222 can subscribe to `acc.>` and read all agent signals, or publish unauthorized signals.

**Solution:** NATS NKeys (NaCl keypairs) provide per-user cryptographic authentication. Each agent role gets a unique NKey seed; the NATS server config lists the corresponding public keys. Agents connect with `nats.connect(nkeys_seed=...)`.

**NKey permission matrix** (once implemented):

| Role | Publish | Subscribe |
|------|---------|-----------|
| `ingester` | `acc.{cid}.task`, `acc.{cid}.heartbeat`, `acc.{cid}.register`, `acc.bridge.{cid}.*.delegate` | `acc.{cid}.task` |
| `analyst` | `acc.{cid}.task`, `acc.{cid}.heartbeat`, `acc.{cid}.alert`, `acc.bridge.{cid}.*.delegate` | `acc.{cid}.task`, `acc.bridge.*.{cid}.result` |
| `arbiter` | `acc.{cid}.heartbeat`, `acc.{cid}.role_approval` | `acc.{cid}.role_update`, `acc.{cid}.alert` |
| `synthesizer` | `acc.{cid}.task`, `acc.{cid}.heartbeat`, `acc.bridge.{cid}.*.delegate` | `acc.{cid}.task`, `acc.bridge.*.{cid}.result` |
| `observer` | `acc.{cid}.heartbeat` | `acc.{cid}.>` (read-only) |

**Files to change:**
- `acc/config.py` — Add `ACC_NATS_NKEY_SEED` to `_ENV_MAP`
- `acc/agent.py` — Pass `nkeys_seed=` to `nats.connect()`
- `operator/internal/reconcilers/infra/nats.go` — Render per-role NKey user config into NATS ConfigMap
- `operator/internal/reconcilers/collective/agent_deployment.go` — Mount NKey Secret into pods

**Note:** Phase 0c can only be properly scoped now that ACC-9 (cross-collective bridge) has defined the full bridge subject schema (`acc.bridge.{from}.{to}.delegate`, etc.). The permission matrix above includes bridge subjects.

## Phase 1 — L7 / eBPF NetworkPolicy ✅ Implemented (opt-in)

> Designed and delivered by **proposal 014** ("L7 / eBPF NetworkPolicy
> for ACC agents"). This section was originally a Cilium-centric sketch;
> proposal 014 superseded it after finding that **Cilium is not the
> default CNI in any ACC deploy scenario** (OpenShift/RHOAI and
> MicroShift default to OVN-Kubernetes; K3s uses Flannel). See
> [`docs/network-policy.md`](./network-policy.md) for the operator guide.

**Problem:** ACC agent pods have no network isolation — any pod that can
route to `NATS:4222` or `Redis:6379` reads every agent signal. NKeys
(Phase 0c) authenticate the NATS *connection*; this phase controls which
pods may *reach* the bus at all.

**Solution — the capability-tiered model.** The operator emits the
highest-tier policy the running cluster's CNI can enforce, capped by
`spec.networkPolicy.maxTier`:

| Tier | Mechanism | Substrate |
|---|---|---|
| 0 | none (no-op) | standalone / no Kubernetes |
| 1 | standard K8s `NetworkPolicy` (L3/L4) | any policy-enforcing CNI (OVN-Kubernetes) — **the committed must-have** |
| 2 | FQDN egress: OVN `EgressFirewall` *or* Cilium `CiliumNetworkPolicy` | OVN-Kubernetes *or* Cilium |
| 3 | full L7 (HTTP method/path) | Cilium only |

Cilium is **one optional Tier-2/3 backend**, never a prerequisite.
"eBPF" is the mechanism Cilium/Tetragon implement — ACC consumes
eBPF-backed engines, it does not write its own.

**Opt-in.** `spec.networkPolicy.enabled` defaults to `false` — upgrading
the operator never drops traffic. `mode: audit` emits the policies
without the default-deny (a safe canary) before `mode: enforce`.

**Per-deploy-mode behaviour:**

- **standalone** — Tier 0 no-op; `NetworkPolicyReady=True/NotApplicableStandalone`.
- **edge-MicroShift** / **rhoai** — OVN-Kubernetes enforces Tier 1; Tier 2
  via `EgressFirewall`; Tier 3 only if Cilium is installed.
- **edge-K3s** — Flannel does **not** enforce `NetworkPolicy`; the
  operator still emits the objects but reports
  `NetworkPolicyReady=False/CNIDoesNotEnforce` rather than give false
  assurance.

**Files (delivered by proposal 014):**
- `operator/internal/reconcilers/security/networkpolicy.go` — the sub-reconciler
- `operator/internal/reconcilers/security/networkpolicy_rules.go` — Tier 1 rule builders
- `operator/internal/reconcilers/security/fqdn_egress.go` — Tier 2 backends
- `operator/internal/reconcilers/security/l7_policy.go` — Tier 3 Cilium L7
- `operator/internal/util/discovery.go` — Cilium / OVN capability detection
- `operator/api/v1alpha1/agentcorpus_types.go` — `NetworkPolicySpec`

## Phase 2 — SPIFFE/SPIRE mTLS 🔲 Planned

**Problem:** Agent identity is a random UUID generated at startup. An agent that restarts gets a new identity. There is no cryptographic proof that a pod claiming to be `analyst` in `sol-01` actually is.

**Solution:** SPIFFE/SPIRE provides stable X.509 workload identity (SVIDs) with automatic rotation. Each agent pod receives a short-lived certificate with a SPIFFE URI encoding its corpus, collective, role, and stable agent ID suffix.

**SPIFFE ID format:**
```
spiffe://{trust_domain}/acc/{corpus_name}/{collective_id}/{role}/{agent_id_suffix}
```
Example: `spiffe://acc.corp/acc/my-corpus/sol-01/analyst/9c1d`

**What SPIRE provides:**
- SPIRE Server (cluster-scoped): issues SVIDs, stores registration entries, talks to K8s attestor
- SPIRE Agent (DaemonSet): node-level workload attestor, serves Workload API socket
- Registration entries: one per `(namespace, serviceAccountName, podLabel)` tuple
- SVIDs rotate automatically (default 1-hour TTL)

**Stable `agent_id`:** The operator stamps `acc.redhat.io/agent-id: {role}-{4-char-hex}` on each agent Deployment. Agents read `ACC_AGENT_ID` from a `fieldRef` label injection. This gives stable identity across pod restarts — a breaking change from the current random UUID.

**NATS TLS with SVID:**
```
tls {
  cert_file: /run/spiffe/tls.crt
  key_file:  /run/spiffe/tls.key
  ca_file:   /run/spiffe/bundle.crt
  verify:    true
}
```

**Redis mTLS:**
- Redis config: `tls-port 6380`; agents connect on port 6380 with SVID client cert
- Adds per-role ACL (`requirepass` stays on port 6379 for backward compat)

**Files to create:**
- `operator/internal/reconcilers/security/spire.go` — Creates `SpiffeID` CRs or calls SPIRE Server API
- `operator/internal/reconcilers/infra/nats.go` — TLS config generation (extend existing file)
- `operator/internal/reconcilers/infra/redis.go` — TLS config + port 6380 (extend existing file)
- `operator/internal/reconcilers/collective/agent_deployment.go` — `spiffe-helper` sidecar injection
- `acc/agent.py` — Read stable `ACC_AGENT_ID`; mTLS NATS connect

**Prerequisite:** SPIRE Server + DaemonSet installed. The operator detects via `spire.spiffe.io/v1alpha1` API group. If absent, `SpireReady=False` condition is set; operator falls back to NKeys only (Phase 0c).

**SPIRE is NOT installed by the ACC operator** — it is a cluster prerequisite.

**Edge mode:** SPIRE is not available at edge. Edge uses NKeys (Phase 0c) + manual TLS certificates via `spiffe-helper` with a self-signed CA (Phase 4).

## Phase 3 — Runtime-evidence Cat-A ✅ Implemented (opt-in)

> Designed and delivered by **proposal 015** ("Runtime-evidence Cat-A").
> This section was originally a Tetragon-centric sketch with two flawed
> premises — see [`docs/runtime-evidence.md`](./runtime-evidence.md) for
> the operator guide.

**Two corrections to the original sketch.** First, Cat-A is **already
real** — `acc/governance.py` has a 3-mode `CatAEvaluator` (subprocess
`opa eval` on the 21-rule `constitutional_rhoai.rego`); the "returns
`True` unconditionally" claim was false. Second, the sketch named
Tetragon as *the* mechanism — but Tetragon is not Cilium-coupled, and
more importantly ACC should not bake in one ecosystem tool.

**Problem:** Cat-A judges *signal metadata the agent process itself
supplies* — forgeable by a compromised or drifted agent. Phase 3 adds a
**kernel-level evidence source the agent cannot falsify.**

**Solution — provider-agnostic.** ACC *consumes* whichever
runtime-security tool the cluster runs. The operator detects the
backend; a single operator-deployed bridge (`acc-runtime-evidence-bridge`)
normalises events into a `KERNEL_EVENT` NATS signal; each agent's
`CognitiveCore` folds events about its own pod into Cat-A.

| Evidence | Source(s) | Hook |
|---|---|---|
| process exec / file open | RHACS (preferred) · Falco · Tetragon | `execve`, `openat` |
| outbound network connect | OpenShift NetObserv (eBPF flow logs, OVN-aware) | `connect` |

**Observe-only first:** `ACC_RUNTIME_ENFORCE=false` (default) — a
4-week observe baseline; kernel violations log as `OBSERVED:kernel:*`
but never block. Then `enforce: true` → `BLOCK:kernel:*` + `ALERT_ESCALATE`.

**Detection — the `cilium.io` collision:** Tetragon's `TracingPolicy`
CRD shares the `cilium.io` group with the Cilium CNI (Phase 1), so
Tetragon is detected at the **resource level** (`TracingPolicy` kind),
never by the group.

**Per deploy mode:** rhoai full; edge opt-in where a backend is
detected; standalone N/A (no privileged-DaemonSet eBPF attach). Cat-A
subprocess evaluation remains the always-available fallback.

**What runtime evidence is NOT used for:** NATS subject-level ACL
enforcement. Kernel/flow evidence sees raw TCP, not NATS protocol
messages. Use NATS NKeys (Phase 0c) for subject ACLs.

## Phase 4 — Hardened Standalone (Podman) 🔲 Planned

**Problem:** Phases 1–3 are Kubernetes-only. The standalone Podman mode has a separate hardening track.

**What is already done:**
- Redis password via `.env` (`ACC_REDIS_PASSWORD`) — implemented in Phase 0b
- Ed25519 ROLE_UPDATE verification — implemented in Phase 0a

**What is planned:**

| Control | Status | Notes |
|---------|--------|-------|
| NATS NKeys | Planned | Same NKey seed approach as Phase 0c; `nats.conf` generated from `.env` file |
| Redis `requirepass` + TLS | Planned | Self-signed CA; `tls-port 6380`; spiffe-helper binary |
| Manual X.509 certificates | Planned | `spiffe-helper` + self-signed CA for mTLS between NATS, Redis, agents |
| Tetragon | **Not available** | eBPF not attachable in standard Podman containers |
| Cilium | **Not available** | Requires Kubernetes |
| SPIRE | **Not available** | Manual `spiffe-helper` + self-signed CA instead |

**Files to change (Phase 4):**
- `deploy/podman-compose.yml` — NKey + Redis TLS volumes
- `deploy/Containerfile.redis` — `tls-port 6380`, `requirepass ${REDIS_PASSWORD}`, `tls-cert-file`
- `deploy/podman-compose.yml` — `spiffe-helper` sidecar pattern with shared volume
- `acc-config.yaml.example` — Document all new security fields

## Implementation Sequence

The dependencies between phases are:

```
Phase 0a — Ed25519 ROLE_UPDATE verify    ← DONE
Phase 0b — Redis password auth           ← DONE
Phase 0c — NATS NKeys                   ← start next (depends on ACC-9 bridge
│                                          subject schema — now available)
│
├── Phase 1 — Cilium NetworkPolicy       ← depends on ACC-9 (bridge subjects →
│                                          port 7422 egress rule for edge)
│
├── Phase 2 — SPIFFE/SPIRE mTLS          ← depends on Phase 0c (NKeys → SPIRE
│     │        stable agent_id           │  replaces NKeys progressively)
│     └── Phase 4 — Hardened Standalone  ← parallel; same concepts, no SPIRE
│
└── Phase 3 — Tetragon Cat-A             ← parallel with Phase 2;
                                           requires Tetragon installed
```

**Recommended order for next work:**
1. **Phase 0c (NATS NKeys)** — highest risk reduction per effort; no infrastructure prerequisites; ~1 week
2. **Phase 1 (Cilium NetworkPolicy)** — standard `NetworkPolicy` fallback makes this low-risk; ~2 weeks
3. **Phase 4 (Standalone hardening)** — parallel with Phase 1; self-signed CA setup
4. **Phase 2 (SPIRE)** — most complex; ~5 weeks; requires SPIRE Server installed
5. **Phase 3 (Tetragon)** — final; ~5 weeks; observe-only mode first
