# Role-definition sync: files ↔ CRDs

ACC stores the *same logical role definition* in two places:

1. **Files** — `roles/<id>/role.yaml` on the agent's local disk.
   Read by `acc.role_loader` at startup and hot-reloaded via a
   file watcher (proposal 003 PR-3).
2. **CRDs** — `AgentCollective.spec.roleDefinition` in the
   Kubernetes API.  Read by the operator reconciler, projected into
   a ConfigMap, mounted into every agent pod.

Before proposal 010 these two surfaces never synchronised: an
operator who edited `role.yaml` saw no change in `oc get
agentcollective`, and an operator who did `oc edit` saw no change in
the file on disk.

Proposal 010 introduces a single configuration flag —
`role_sync.role_source` — that picks which side is authoritative and,
optionally, runs both directions at once with last-writer-wins
reconciliation.

---

## The three modes

| `role_source` | Default for `deploy_mode` | Files | CRDs |
|---|---|---|---|
| `files` | `standalone` | **source of truth** | derived projection (written by the operator reconciler) |
| `crd` | `rhoai` | derived projection (written by the agent's `RoleCRDProjector`) | **source of truth** |
| `mirror` | `edge` | bidirectional | bidirectional |
| `auto` | (resolved at validation time) | — | — |

`auto` is the implicit default — it resolves to one of the three
concrete values based on `deploy_mode`.  Operators can override
explicitly via `acc-config.yaml`:

```yaml
role_sync:
  role_source: mirror
  conflict_window_s: 2.0
  events_subject: acc.role.sync
```

Environment overrides: `ACC_ROLE_SOURCE`,
`ACC_ROLE_SYNC_CONFLICT_WINDOW_S`, `ACC_ROLE_SYNC_EVENTS_SUBJECT`.

## Mode semantics

### `files`

```
roles/<id>/role.yaml ──fsnotify──▶ operator file watcher
                                   │
                                   ▼
                          patch AgentCollective CR
                                   │
                                   ▼
                          ConfigMap → pod mount
```

The operator binary runs with `--role-source=files` (or
`ACC_ROLE_SOURCE=files`).  Files on disk are authoritative.  Any edit
in the operator's `oc edit` or OpenShift Web Console is overwritten
on the next file event.

Use for: standalone laptop dev, edge deployments where the
operator's view of the world is a git-checked-out `roles/` tree.

### `crd`

```
oc edit / Web Console ──▶ AgentCollective spec.roleDefinition
                                   │
                          (every 30s poll by agents)
                                   │
                                   ▼
                    RoleCRDProjector writes roles/<id>/role.yaml
                                   │
                                   ▼
                    RoleLoader file watcher → agent hot-reload
```

The agent runs with `ACC_ROLE_SOURCE=crd` (or via
`acc-config.yaml`).  The `kubernetes` Python package must be
installed; otherwise the projector fails fast with a clear error.

CRDs are authoritative.  Any file edit on disk gets overwritten the
next time the projector polls (default every 30 s).  Generated files
carry a sentinel header naming the CRD they came from:

```yaml
# Generated by acc.role_crd_loader from CRD
# AgentCollective: default/coding_agent
# Source-of-truth: CRD (role_sync.role_source=crd or mirror).
# Do not edit this file by hand — set role_source=files in
# acc-config.yaml to make local files SOT instead.
# Last projected: 2026-05-14T15:23:11Z
role_definition:
  purpose: Generate code
  …
```

Use for: OpenShift production where the operator's only contact
with the cluster is through the K8s API.

### `mirror`

Both directions run simultaneously.  Conflict detection (proposal
010 PR-4) prevents echo loops: every CRD-driven file write is
recorded, and the next file-watcher event for the same role is
classified:

| Within `conflict_window_s` | Content matches our record | Classification |
|---|---|---|
| yes | yes | **echo** (silent — our own write coming back through the watcher) |
| yes | no | **conflict** (concurrent file + CRD write — LWW resolution) |
| no | — | **applied** (genuine operator edit, propagate to CRD) |

Conflicts publish on NATS subject `<events_subject>.conflict` with a
payload like:

```json
{
  "role_id": "coding_agent",
  "winner_source": "crd",
  "loser_source": "file",
  "loser_snippet": "role_definition:\n  purpose: was-edited-locally\n",
  "ts": 1747237391.0
}
```

The TUI Ecosystem screen subscribes to this subject and renders a
small red badge in the role detail header when there's a fresh
conflict (default window 5 minutes).

Use for: laptop test environments where the operator legitimately
edits both surfaces during development.

## Sequence diagram — `mirror` mode echo handling

```
Operator        File           Operator         Agent           NATS
                System       (Go controller)  (RoleCRDProjector)
   │             │                │                │              │
   │ oc edit AC  │                │                │              │
   │─────────────────────────────▶│                │              │
   │             │                │ patch AC       │              │
   │             │                │   spec         │              │
   │             │                │ (CRD updated)  │              │
   │             │                │                │ poll (30s)   │
   │             │                │                │◀─────────────│
   │             │ write role.yaml│                │              │
   │             │◀───────────────│                │              │
   │             │      (echo)    │                │              │
   │             │ fsnotify event │                │              │
   │             │────────────────▶                │              │
   │             │                │ ParseRoleFile  │              │
   │             │                │ → identical    │              │
   │             │                │ → no-op patch  │              │
   │             │                │                │              │
   │             │                │                │ publish      │
   │             │                │                │ .applied     │
   │             │                │                │─────────────▶│
   │             │                │                │              │
```

The `ParseRoleFile`-based no-op check on the Go side AND the
sentinel-strip content comparison on the Python side together
guarantee the echo dies in one round trip.

## Operator runbook

### Switch a host from `files` to `crd`

1. Confirm the CRD already carries the role definitions you want as
   SOT.  `oc get agentcollective <name> -o yaml | grep -A20
   roleDefinition` should show non-empty fields.
2. Stop the operator-binary with `--role-source=files` (or your
   agents that have `ACC_ROLE_SOURCE=files`).
3. Run one cycle of `mirror` mode to validate both sides agree:

   ```bash
   ACC_ROLE_SOURCE=mirror acc-agent --collective-id sol-01
   # Watch NATS for one minute:
   nats sub 'acc.role.sync.>'
   # Expect only .applied events; zero .conflict.
   ```

4. Flip to `crd` once `mirror` shows no conflicts.

### Switch a host from `crd` to `files`

Symmetric: same `mirror`-validation step in the middle.

### Diagnose a stuck conflict badge

The TUI badge stays red for `badge_window_s` (default 300 s) after
the last conflict event.  If you keep seeing red:

1. Check the conflict counter: `nats req '_INBOX' '' …` or read the
   listener's `conflict_count` via the TUI Configuration screen
   (PR-5 wires this).
2. Look for a fight between two editors — typically your terminal +
   the OpenShift Web Console on the same role.
3. Pick one as the winner: switch the other side to read-only by
   flipping `role_source` to that side's name.

## Non-goals

This proposal explicitly does **not** address:

- **`role.md` sync** — narrative authoring stays on the file side.
  The CRD has no field for it; cross-host `role.md` distribution is
  a git-or-OCI problem.
- **Three-way merge** — only LWW.  Smarter conflict resolution can
  come later.
- **Spiffe-style role identity** — proposal 004 territory.  Today
  the join key is plain `role.id` (matches both filesystem
  directory name and `metadata.name` on the CR).
- **PD extension edit path** — the PD extension subscribes to the
  same `acc.role.sync.*` NATS events but doesn't write files or CRDs.
